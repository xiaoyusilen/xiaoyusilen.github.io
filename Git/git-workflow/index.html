Git Workflow

2017-03-07 • 分类: Git • 标签: Git

什么是Git workflow

  • workflow叫做工作流程，”flow”原意是水流，比喻项目像水流那样，顺畅自然地向前流动，不会发生冲击、对撞、甚至旋涡。
  • Git作为一个源代码管理系统，不可避免涉及到多人协作，多分支管理，很容易出现冲突的问题。

介绍三种广泛使用的工作流程：

  • Git flow
  • Github flow
  • Gitlab flow

这三种工作流程，有一个共同点就是：都采用”功能驱动式开发”（Feature-driven development）

Git flow

  • 由Vincent Driessen（nvie.com）在2010年提出
  • 文章：A successful Git branching model（http://nvie.com/posts/a-successful-git-branching-model/）

GitFlow的几个分支

  • main branches
  • master：存放的一定是可以直接拿去部署到生产环境的稳定版本，应该和当前生产环境的代码是一致的，有且只有一个主分支。
  • develop：存放的是每日构建的集成分支，到达稳定状态时可以发布并merge回master
  • develop一般是从master分支check出来的

  • supporting branches
  • feature：每次开发新的功能都应该新建一个feature/xxx分支，在独立的feature分支上进行开发，开发完毕后，code review通过后，merge回develop分支
  • release：为每次发布准备的release candidate，在这个分支上只进行bug fix，并在完成后merge回master和develop
  • hotfix：用于快速修复，在修复完成后merge回master和develop

GitFlow的优缺点

  • 思考：为什么开发一个新的feature非得新开一个branch，而不是直接在develope上进行？
  • 最根本的问题在于Github背后的一套feature branch模型。
  • feature branch模型本质上试一种代码隔离技术，使用feature branch通常的做法是：当developer开始一个新feature，基于develop branch的最新代码建立一个独立branch，然后在branch上完成feature开发，开发不同feature上的developer因为工作在彼此隔离的branch上，因此相互之间的工作不会有影响，直到feature开发完成，将feature branch上的代码merge回develop branch。
  • merge代码是痛苦且易错的，虽然所有的git工作流都会建议我们频繁地从master pull代码，早做merge，然而feature branch这个实践本身阻碍了频繁的Merge，因为不同feature branch只能从master或者develop分支pull代码，而较长周期的开发完成后才被merge回master，也就是说相对于不同的feature branch，develop上的代码永远是过时的，如果feature开发的平均时间是一个月，feature A所基于的代码可能在一个月前已经被feature B修改了，这一个月来一直是基于错误的代码进行开发，而直到feature branch B被merge回develop才能获得反馈，到最后merge的成本是非常高的。
  • 各个feature之间的代码是隔离的，可以独立地开发、构建、测试；
  • 当feature的开发周期长于release周期时，可以避免未完成的feature进入生产环境；

  • git虽然非常方便，但也会产生副作用，很可能会留下一个枝节蔓生、四处开放的版本库，到处都是分支，完全看不出主干发展的脉络。
  • 而git flow则是一个有效的分支管理策略，使得版本库保持简洁，主干清晰，各个分支各司其职，井井有条。

Git使用Tips

  • 开发的时候经常会遇到多次commit的情况，应用git rebase可以帮助合并多次commit，git rebase origin/master，具体操作这里就不说了，但是非常好用，开发的时候有时候切换分支非常麻烦需要commit，但是push代码的时候多次提交又非常的不好看，适当的利用git rebase会帮助我们管理commit记录。

  • 关于开发流程，我的建议是，拿到一个新的需求的时候，作为开发负责人，应该在相应的项目下新建一个issue，在issue内写明负责人、需求背景，关联相关的需求文档，以及接口设计等，并且创建WBS，合理的利用issue的标签。

  • 这样，一个完整的需求issue中就可以看到这个需求实现的完整过程，从提出需求的需求文档，到接口设计，数据库设计修改，讨论的问题，最终的结果，以及中间代码修改的merge request，一目了然。
