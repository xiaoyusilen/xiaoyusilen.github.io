这么看来好像只是为了排序用的，这时候我仔细又回去读了一遍Interface的注释：

A type, typically a collection, that satisfies sort.Interface can be sorted by the routines in this package. The methods require that the elements of the collection be enumerated by an integer index.

划重点，这特么几个意思，意思就是，我们在源码sort.go里看到的一切返回为Interface的方法都只是为了给sort.Sort()用的，你可以作为Sort()的参数，然后没有然后了….

So when you write sort.Reverse(sort.IntSlice(s)), whats happening is that you’re getting this new, ‘modified’ IntSlice that has it’s Less method replaced. So if you call sort.Sort on it, which calls Less, it will get sorted in decreasing order.

尼玛，在这里才调用了Less()？不用Sort的话，Less()就不会被调用，这里是真的心机。

起初是不明白为什么这样设计的，直到我问了大佬之后恍然大悟，其实Sort()我们不能单纯理解为正向排序，而是一个抽象到具体的过程，具体如何排序取决于我们传入的interface中究竟带了一个什么样的排序算法，Sort()只是一个执行的过程。所以我们在调用Reverse之后其实是生成了一个翻转排序的算法，然后扔给Sort()去执行了。这样说来，就恍然大悟了。讲真，这样看来这个设计还真是巧妙。

总结

果然还是有很多设计，我们只看到表象而不知道为什么要这样去做，而官方的文档和注释又不够多，所以会给人以一定的误导，这时候大佬曰就显得格外重要了。吐槽一下Go的文档，恩，就这样，Thx~
